## Recorded-Objection Discipline (ROD) Exercise

**ADR ratified over blocking objection: ADR-0050**

Per CONSTITUTION.md Section 3, a blocking objection may not be dismissed without
a recorded rationale. This run exercised the ROD once on ADR-0050, the
decision that establishes `authenticatedFetch()` in `src/config/remotes.ts` as
the sole module constructing authenticated outgoing HTTP requests.

### Objecting Party

landvault-backend maintainers

### Substance of the Concern

> The explicit `token: string` parameter on `authenticatedFetch()` forces every
> call site in every service file to thread the token through its call stack. At
> scale, this produces significant boilerplate: every service method must accept
> and forward a token parameter even if it does nothing else with it. A
> module-level auth accessor (reading the current token from a shared singleton)
> would eliminate this per-call-site burden without compromising the
> single-construction-point invariant.

This is a blocking concern because every backend-access service in the shell and
in pod repos is affected; teams adopting the pattern would inherit the threading
boilerplate.

### Dismissal Rationale

The concern identifies a genuine ergonomic cost but does not prevent ratification
of ADR-0050 as written, for three reasons:

1. **Separability.** The immediate goal ADR-0050 addresses — eliminating six
   independent auth-header construction patterns across six files — is separable
   from the token-threading ergonomics. These are two distinct problems; resolving
   one now does not preclude resolving the other in a follow-on ADR.

2. **Purity and testability.** The explicit token parameter keeps
   `authenticatedFetch()` pure: the function takes its inputs as arguments and
   has no hidden dependencies. A module-global auth accessor introduces ordering
   hazards (singleton read before auth initialises) and test isolation problems
   that require a separate design decision. Those hazards are not hypothetical —
   they are why the pre-ADR-0050 state had multiple ad-hoc header patterns in the
   first place.

3. **Follow-on resolution.** The objecting party's preferred solution
   (DI-injected auth accessor eliminating per-call-site token threading) is
   architecturally sound. It is addressed in full in ADR-0055 (formerly
   DRAFT-ADR-0053), which amends ADR-0050 on exactly this point by introducing
   the pod-sdk service layer and composition-root wiring. Ratifying ADR-0050 now
   and ADR-0055 immediately after satisfies both goals in the correct sequencing.

The platform interest in closing six independent auth-header patterns outweighs
the per-call-site threading cost for the interim period between ADR-0050 and
ADR-0055 ratification. The objection is dismissed. The dismissal is recorded in
ADR-0050's `dismissed_objections` field in LOG.yaml per CONSTITUTION.md §3.1.
