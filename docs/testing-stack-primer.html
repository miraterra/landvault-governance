The Testing Stack

The four layers we test in, what each is for, and when each runs.

Most arguments about testing are really arguments about vocabulary — people mean different things by "integration test," and talk past each other. This sets out the four layers we use, the one question each answers, and — because cost rises as you go up — how often each runs. The contract-testing primer covers one layer in depth; this is the wider map.

In brief

  • Four layers — unit, contract, E2E, UAT — each answers a different question.
  • The first three verify we built it right; UAT validates we built the right thing.
  • "Correctness" is spread across the layers, and the cheap, automatable kinds sit at the bottom.
  • Cost rises up the stack, so frequency falls: every commit → pre-merge → post-deploy → per release.
  • E2E checks you can get there — never whether what's there is correct.
  • There is deliberately no whole-system "is everything correct" test. That's what the lower layers are for.

1The four layers

Each layer answers exactly one question, and they get more expensive as you climb:

LayerThe question it answersAutomated?When it runs
UnitIs each component internally correct? (white-box, in isolation)yesevery commit, liberally
Contract (CCT)Does each boundary honor its agreed interface? (black-box, per boundary)yespre-merge gate
E2ECan a real user get in and reach each part of the deployed system? (black-box, whole system)yespost-deploy smoke

2Verification vs. validation

The split that organizes the stack: the first three layers are verification — "did we build it right?" Each checks a different facet: component logic, contract conformance, wiring. UAT is validation — "did we build the right thing?", which only a human can judge.

So "correct" isn't one thing owned by one layer — it's distributed, and the kinds that can be automated are pushed down to the cheap layers. What's left for UAT is only the irreducibly human part: fitness for purpose. (This is the same logic as encoding contract semantics in tiers and leaving a human-reviewed residue — here applied to the whole system.)

3E2E, specifically

E2E is the layer most likely to be misunderstood, so it gets the sharpest rule. It tests one thing: that the assembled, deployed system is reachable and correctly wired — that a real user can get in and arrive at each part. It does not check whether what they find is correct.

A reachability journey is a test that a logged-in user can get to a given page and it loads without erroring.

And the line that keeps it honest: it checks only that the page loads — never whether the content is correct (that's contract and UAT).

Concretely, one looks like this:

1. log in as the test user
2. open /orders
3. ✓ the page rendered (didn't crash, didn't bounce to login)
4. ✓ the /api/orders call came back 200 (not 401/404/500)
5. stop — do NOT look at what the orders actually say

The intuition: knock on the door, check it opened, walk away. You don't go inside and inspect the furniture. The same page seen through all three automated layers asks three different questions:

  • E2E: can I get to the orders page and does it load?
  • Contract: are the orders themselves right?
  • UAT: is the orders page actually any good for a real user?

When unsure whether something is E2E's job, ask: could a per-boundary test, against one independently-deployable boundary, catch it? If yes, it isn't E2E. If it only shows up once everything is wired and deployed together — a real login actually landing you authenticated, a route resolving, a pod mounting — it's E2E.

4Cost drives cadence

Cost rises as you climb, so frequency falls. Unit tests are cheap and run on every commit. Contract tests gate a merge. E2E needs a deployed environment, so it runs after a deploy to staging, as a thin smoke check. UAT costs human time and runs per release.

E2E is detection, not prevention. Because it runs post-deploy, it tells you staging broke — it doesn't stop the bad merge. Prevention lives in the layers below it (unit and contract, pre-merge). E2E is the last "did the wiring actually hold in a real environment?" check before staging is promoted.

5The deliberate gap

There is no automated whole-system behavioral test — on purpose. Behavior is decomposed down to unit and contract, where it's fast and clearly owned. Good contract testing is exactly what lets E2E stay a thin wiring check and UAT stay a small human pass, instead of both bloating into slow, flaky, owner-less regression suites. The absence of that layer is a feature, not a gap.

6Who owns what

Unit belongs to the sub-system owner; contract to whichever side drives it (see the contract primer); UAT to the UAT owner. E2E is the only layer with no single boundary — it spans everything — so its ownership splits:

  • Each pod owner writes the reachability journey to their own pod (they know its entry point and what "loaded" means).
  • The integrator (the shell) owns the shared harness, runs the suite, and triages a failure to the layer that caused it (routing → shell, didn't mount → pod, not deployed → infra).
  • The Testing Practices coordinator owns the practice itself — the shared bar, and cross-cutting concerns like the test user's identity and seed data (which no single layer owns).

7A note on the words

"Integration testing" is used two different ways here, which is exactly the kind of ambiguity that causes the talking-past-each-other this primer opens with. The Testing Practices charter uses "integration testing" for the whole-system layer (what we call E2E); the contract ADRs use it for the per-boundary layer (what we call contract / CCT). To avoid the collision, prefer the canonical names: unit, contract, E2E, UAT.

This is the wider-map companion to the contract-testing primer. The binding source of truth is governance ADRs 0063–0065 (the stack, the E2E scope rule, E2E ownership) in the landvault-governance corpus. Where this primer and the ADRs disagree, the ADRs win — and flag it, so we can fix the primer.
